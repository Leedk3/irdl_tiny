1. class와 object
    - class: object를 만들기 위한 설계도 또는 툴
    > 실제로 존재하지 않는 관념적인 규칙 메모리 차지x, 변수와 함수를 정의만 해둠

    - object: class 설계도를 바탕으로 실제로 만들어진 실체 
    > 컴퓨터의 메모리 공간을 차지o, class에 적힌 대로 데이터 값을 실제로 갖고 있음 

2. class와 object의 컵셉 비교
    - 은닉성: 데이터(변수)와 그 데이터를 다루는 방법(함수)을 하나로 묶고 내부의 상세한 구현을 외부로부터 숨김
    > class: 어떤 정보를 private으로 할지, 어떤 기능을 public으로 할지 규칙을 생성 
      ex) 비밀번호는 숨기고, 비밀번호 변경 버튼만 보여주자
    > object: 실제로 생성된 객체는 이 규칙에 따라 자신의 데이터를 보호 외부에서 강제로 데이터를 수정하려고 하면 에러를 발생시켜 객체의 무결성
        *객체의 무결성이란? 객체가 갖고 있는 데이터가 파괴되지 않고 처음부터 끝까지 일관된 상태로 유지

    - 상속: 부모 클래스의 모든 특성을 자식 클래스가 물려받는 것
    > class: "로봇은 움직일 수 있다"는 공통 분모를 부모 클래스에 정의 자식 클래스인 '드론'은 이를 물려받아 처음부터 설계할 필요 없이 바로 '날기' 기능만 추가
    > object: 실제로 만들어진 '드론 객체'는 부모로부터 물려받은 '이동' 능력과 자신의 특기인 '비행' 능력을 모두 메모리에 가지고 활동합니다.

    - 다형성: 다형성은 같은 이름의 명령이 객체에 따라 다르게 동작하는 성질
    > class: 부모 클래스에서 move()라는 이름을 정의하고, 자식 클래스마다 이 함수가 구체적으로 어떻게 작동할지 다르게 프로그래밍(오버라이딩)
    > object: 사용자가 여러 로봇 객체들에게 똑같이 움직이라고 명령했을 때, 바퀴 로봇 객체는 굴러가고 드론 객체는 날아오르는 등 각자의 정체성에 맞는 행동

    - 추상화: 추상화는 복잡한 내부 동작 원리는 숨기고 사용자에게 꼭 필요한 핵심 인터페이스만 제공
    > class: 사용자가 알 필요 없는 복잡한 연산 과정은 내부 함수로 숨기고, 밖에는 start() 같은 단순한 버튼 하나만 설계
    > object: 사용자는 객체가 제공하는 단순한 인터페이스만 사용하여 복잡한 기능을 손쉽게 제어
    
    *부모클래스? 자식클래스?
        > 부모클래스: 상위 클래스로 공통된 기능 제공 = 일반적인 자동차 설계도
        > 자식 클래스: 하위 클래스로 특정 기능 추가 및 확장 = 전기차, 수소차 설계도

3. class와 object 구조 설명
    struct univ >>>>>> univ란 class지정 아래는 클래스에 담긴 변수 
    {
        int established_year;
        int president_name;
    }


    univ JBNU;
    JBNU.established_year = 1945;

    univ KAIST;
    KAIST.established_year = 1960;

    - ros2 custom message정의
        int32 established_year
        string president_name    

    - pakage생성
        #include "univ.hpp" // univ란 생성된 설계도 불러오기

        // 객체 생성 (JBNU라는 실체 만들기)
        auto JBNU = irdl_msgs::msg::Univ(); 
        JBNU.established_year = 1945; 
        JBNU.president_name = "양오봉"; 

        // 객체 생성 (KAIST라는 실체 만들기)
        auto KAIST = irdl_msgs::msg::Univ();
        KAIST.established_year = 1971;
